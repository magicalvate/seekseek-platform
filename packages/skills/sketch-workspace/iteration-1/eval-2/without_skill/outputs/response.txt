已把这个想法记录到 sketch/ideas.md。

核心思路：用 Redis 替换进程内存中的 session 缓存，让所有实例共享同一个会话存储，从根本上消除水平扩展时的状态同步问题。

几个值得提前想清楚的点：

1. **session key 设计**：建议用 `session:{sessionId}` 格式，TTL 与原来的 session 过期时间保持一致，不用单独维护过期逻辑。

2. **序列化**：JSON 最简单，MessagePack 更紧凑省内存，数据量大的话可以考虑后者。

3. **连接管理**：用连接池，不要每次请求都新建连接，否则延迟会放大。

4. **Redis 本身的 HA**：引入 Redis 解决了实例间同步问题，但 Redis 自己变成了新的单点——要配 Sentinel 或 Cluster，否则 Redis 挂了所有实例都丢 session。

5. **显式清理**：session 数据不再随进程退出自动消失，用户注销或强制踢出时要主动 DEL 对应 key。

网络延迟这块通常不是问题，局域网内 Redis 访问一般 <1ms，和进程内内存比差距感知不明显。
